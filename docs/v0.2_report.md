# AgentGym v0.2 Report

Date: 2026-05-17

AgentGym v0.2 is a 10-task executable mini-benchmark for coding agents. It keeps the same public task format from v0.1, but expands the task set and adds suite-level agent execution.

The purpose of this release is still not leaderboard breadth. It is to prove that a small benchmark can run end-to-end across multiple realistic failure modes and report agent, public-test, and hidden-test outcomes from one command.

## Benchmark Snapshot

| Metric | Result |
| --- | --- |
| Tasks | 10 |
| Language | Python |
| Domain | API validation, serialization, state, configuration, datetime, and authorization |
| Public test result on starter tasks | 10/10 pass |
| Hidden test result on starter tasks | 0/10 pass |
| Runner | Local CLI |
| Agent invocation | Optional trusted local command |
| Suite agent command | `agentgym run-suite --agent "<command>"` |

The expected starter baseline is that every task passes public tests and fails hidden tests. Public tests show plausible behavior. Hidden tests check the failure mode the task is designed to isolate.

## Current Tasks

| Task | Primary failure mode | Hidden behavior checked |
| --- | --- | --- |
| `python-api-001` | Input normalization | Whitespace-only required fields are rejected. |
| `python-api-002` | Nested type coercion | String and integer preference values are rejected instead of coerced. |
| `python-api-003` | Cross-field validation | Account type and tax ID rules are enforced together. |
| `python-api-004` | Response serialization leak | Internal user fields are not leaked in API responses. |
| `python-api-005` | Pagination boundary validation | Invalid limits, offsets, numeric strings, blank queries, and unsupported sort values are rejected. |
| `python-api-006` | Partial update semantics | Omitted fields, explicit nulls, empty updates, unknown fields, and boolean coercion are handled correctly. |
| `python-api-007` | Idempotency-key handling | Duplicate payment requests do not overwrite state, double-process, or accept conflicting payloads. |
| `python-api-008` | Config precedence | Layered config sources use the correct precedence and reject invalid overrides instead of falling back. |
| `python-api-009` | Datetime boundary validation | Timezone-aware booking windows reject naive timestamps, equal windows, and UTC ordering mistakes. |
| `python-api-010` | Authorization scope enforcement | Normal users cannot access documents owned by another actor, and unknown actor roles are rejected. |

Each task is intentionally small. The value is in isolating one bug class cleanly enough that a failure can be interpreted.

## Agent Harness

AgentGym can run the same trusted local command once per copied task workspace:

```bash
agentgym run-suite --agent "<command>"
```

The source task directories are not mutated. If the agent command fails on one task, the suite records that failure and continues evaluating the remaining tasks.

Using a no-op command shows the suite-agent reporting shape:

```text
10/10 agent commands passed
10/10 tasks passed public tests
0/10 tasks passed hidden tests
```

That result does not mean the no-op command solved the benchmark. It means the harness can execute an agent command across the suite while preserving the public/hidden scoring split.

## Limitations

AgentGym v0.2 is still an open public method and harness proof. It does not yet provide:

- private held-out tasks
- contamination-resistant evaluation
- Docker or sandbox isolation
- model registry
- leaderboard scoring
- broad language or framework coverage

The hidden tests are public in this repository. That makes the tasks useful for development, smoke testing, and methodology work, but not yet for final model ranking.

## Next Milestones

The next useful milestones are:

1. Try real coding-agent commands against the 10-task suite.
2. Add a small result summary that groups failures by task and primary failure mode.
3. Decide what a private or generated task pack should look like.
4. Keep the public tasks as the inspectable reference format.

The constraint should stay the same: every task must be executable, deterministic, focused on one failure mode, and backed by a reference patch.
